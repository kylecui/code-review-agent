## 1. Executive Summary

v0.2.0 transforms Agent Review from a 4-collector code review agent into a multi-engine fusion SAST platform. The upgrade introduces:

- **8 new collectors** across 3 engine tiers (L1 pattern matching, L2 deep semantic, L3 graph-based)
- **Universal SARIF adapter** for standardized cross-engine data interchange
- **Enhanced AST-aware fingerprinting** for refactor-resilient dedup
- **Engine tier routing** that maps scan tracks (PR incremental vs. baseline full) to appropriate engine sets
- **Reachability analysis** correlating SCA dependency vulns with actual code call paths
- **Expanded language support**: Java, Go, Python, C/C++, **C#, JavaScript, TypeScript, Lua** (13+ languages total)
- **SaaS multi-tenancy**: Per-user isolated repo configurations and LLM API keys with strict tenant boundaries

### Design Principles Carried Forward

1. **Backward compatible** — existing 4 collectors, normalizer, pipeline, tests all remain unchanged
2. **Same extension pattern** — new collectors inherit `AbstractCollector`, new normalizers are `_normalize_<tool>()` methods
3. **Additive-only DB changes** — new nullable columns and tables, no migrations breaking existing data
4. **Incremental rollout** — each phase is independently deployable and testable
5. **Strict tenant isolation** — per-user settings and repos are never shared across users

### 4.1 GitleaksCollector (`collectors/gitleaks.py`)

**Tier**: L1 (Pattern Matching) — complements the existing `SecretsCollector` (GitHub API-based) with local CLI-based secret detection.

**Tool**: [Gitleaks](https://github.com/gitleaks/gitleaks) — scans git repos and files for hardcoded secrets.

**Design**:
```python
class GitleaksCollector(AbstractCollector):
    name: ClassVar[str] = "gitleaks"

    def __init__(self, settings: Settings):
        self._settings = settings

    async def collect(self, context: CollectorContext) -> CollectorResult:
        # 1. Skip if gitleaks_mode == "disabled"
        # 2. Determine scan target: context.local_path or download repo
        # 3. Run: gitleaks detect --source <dir> --report-format sarif --report-path <tmpfile> --no-git
```

**Severity Mapping**:

## 5. SARIF Adapter

### Purpose

Provide a universal translator from SARIF v2.1.0 to the internal `raw_findings` format used by `FindingsNormalizer`. Many tools now output SARIF natively (Gitleaks, golangci-lint, CodeQL, SpotBugs with plugins). This eliminates the need for per-tool JSON parsers.

### File: `src/agent_review/normalize/sarif_adapter.py`

### Design

```python
@dataclass(slots=True)
class SarifFinding:
    """Intermediate representation from SARIF result."""
    rule_id: str
    path: str
    line: int
    end_line: int | None
    severity: str  # SARIF level: error/warning/note/none
    message: str
    snippet: str
    fingerprint: str  # from SARIF partialFingerprints or computed
    category: str  # from rule.properties.tags or rule.defaultConfiguration
    cwe: list[str]
    precision: str  # from rule.properties.precision
    tool_name: str  # from run.tool.driver.name


class SarifAdapter:
    """Converts SARIF v2.1.0 JSON to list of raw_findings dicts."""

    def parse_file(self, sarif_path: Path) -> list[dict[str, object]]:
        """Read a SARIF file and return raw findings."""

    def parse_json(self, sarif_data: dict[str, object]) -> list[dict[str, object]]:
        """Parse a SARIF JSON dict and return raw findings."""

    def _extract_runs(self, sarif: dict) -> list[dict]:
        """Extract all runs from SARIF envelope."""

    def _extract_results(self, run: dict) -> list[SarifFinding]:
        """Extract results from a single run, resolving rule metadata."""

    def _resolve_rule(self, run: dict, rule_id: str, rule_index: int | None) -> dict:
        """Look up rule in run.tool.driver.rules or run.tool.extensions."""

    def _extract_location(self, result: dict) -> tuple[str, int, int | None]:
        """Extract file path, start line, end line from result.locations."""

    def _extract_fingerprint(self, result: dict) -> str:
        """Extract from partialFingerprints or compute fallback."""

    def _extract_cwe(self, rule: dict) -> list[str]:
        """Extract CWE references from rule.properties.cwe or tags."""

    def _map_severity(self, level: str) -> str:
        """Map SARIF level to our severity string."""
```

### SARIF → raw_findings Mapping

| SARIF Field | raw_findings Key |
|---|---|
| `result.ruleId` / `result.rule.id` | `rule_id` |
| `result.locations[0].physicalLocation.artifactLocation.uri` | `path` |
| `result.locations[0].physicalLocation.region.startLine` | `line` |
| `result.locations[0].physicalLocation.region.endLine` | `end_line` |
| `result.level` | `severity` (mapped to ERROR/WARNING/INFO) |
| `result.message.text` | `message` |
| `result.locations[0].physicalLocation.region.snippet.text` | `snippet` |
| `result.partialFingerprints.*` | `fingerprint` |
| `run.tool.driver.rules[].properties.tags` (filtered for "security", "correctness", etc.) | `category` |
| `run.tool.driver.rules[].properties.cwe` | `cwe` |
| `run.tool.driver.rules[].properties.precision` | `confidence` |
| `run.tool.driver.name` | `tool_name` (metadata) |

### Integration with Normalizer

The `FindingsNormalizer` gains a `_normalize_sarif_based()` shared method that any SARIF-producing collector can use:

```python
# In FindingsNormalizer
def _normalize_sarif_based(self, result: CollectorResult, tool_prefix: str) -> list[FindingCreate]:
    """Shared normalization for SARIF-based collectors (gitleaks, golangci_lint, codeql, etc.)."""
    sarif_adapter = SarifAdapter()
    # result.metadata["sarif_data"] or result.raw_findings already parsed
    ...
```

Collectors that use the SARIF adapter will store SARIF data in `result.metadata["sarif_data"]` or pre-parse it into `raw_findings` via the adapter before returning.

**Recommended approach**: Each collector calls `SarifAdapter.parse_file()` internally and returns normalized `raw_findings` in the same dict format as existing collectors. The normalizer dispatch remains by `collector_name` for maximum control.

## 6. Enhanced Fingerprinting

### Current (v0.1.0)
```python
fingerprint = SHA256("tool|rule_id|path|line")
```
**Problem**: Line-based fingerprints break when code is moved (refactored) but unchanged.

### Enhanced (v0.2.0)
```python
fingerprint = SHA256("rule_id|file_path|normalized_ast_snippet")
```

### Design: `src/agent_review/normalize/fingerprint.py`

```python
class FingerprintGenerator:
    """Generate refactor-resilient fingerprints for findings."""

    def compute(self, rule_id: str, file_path: str, snippet: str, line: int) -> str:
        """
        Compute fingerprint with AST normalization.

        Strategy (cascading fallback):
        1. If snippet available: normalize whitespace, strip comments, hash
        2. If SARIF partialFingerprints present: use directly
        3. Fallback: SHA256("rule_id|file_path|line") (v0.1.0 compat)
        """

    def _normalize_snippet(self, snippet: str) -> str:
        """
        Normalize code snippet for fingerprinting:
        - Strip leading/trailing whitespace
        - Collapse internal whitespace runs
        - Remove single-line comments (// and #)
        - Remove blank lines
        - Lowercase identifiers? NO — keep case to avoid false merges
        """

    def _hash(self, canonical: str) -> str:
        return hashlib.sha256(canonical.encode("utf-8")).hexdigest()
```

### Migration Strategy

- v0.2.0 computes **both** old-style and new-style fingerprints
- `FindingCreate` schema gains an optional `fingerprint_v2: str | None` field
- `FindingsDeduplicator` first attempts dedup on `fingerprint_v2`, falls back to `fingerprint`
- Existing suppressions (by fingerprint) remain valid
- After one full baseline scan, new fingerprints are established

## 8. Reachability Analysis Module

### Purpose

Correlate SCA (Software Composition Analysis) dependency vulnerability data with SAST code-call data. Only flag a dependency vulnerability as actionable when the code actually **calls the vulnerable function** in the affected library.

### File: `src/agent_review/normalize/reachability.py`

### Design

```python
@dataclass(slots=True)
class ReachabilityResult:
    reachable_findings: list[FindingCreate]  # dependency vulns confirmed reachable
    unreachable_findings: list[FindingCreate]  # dependency vulns with no call path evidence
    total_filtered: int  # how many were demoted

class ReachabilityAnalyzer:
    """
    Cross-reference dependency vulnerability findings with code analysis findings
    to determine if vulnerable code paths are actually reachable.
    """

    def analyze(
        self,
        findings: list[FindingCreate],
        sca_findings: list[FindingCreate] | None = None,
    ) -> list[FindingCreate]:
        """
        Filter/annotate findings based on reachability.

        Strategy:
        1. Separate SCA findings (category startswith "dependency.") from SAST findings
        2. For each SCA finding, check if any SAST finding references the same library/function
        3. If reachable: keep severity
        4. If NOT reachable: demote severity by 1 level, add "(unreachable)" annotation
        5. Return merged list
        """

    def _is_sca_finding(self, finding: FindingCreate) -> bool:
        return finding.category.startswith("dependency.") or finding.category.startswith("sca.")

    def _find_call_evidence(
        self,
        sca_finding: FindingCreate,
        sast_findings: list[FindingCreate],
    ) -> bool:
        """
        Heuristic: Check if any SAST finding's file_path imports or references
        the package/module mentioned in the SCA finding's evidence.
        """
```

### Scope for v0.2.0

**Phase 1 (v0.2.0)**: Heuristic-based — match SCA vulnerability package names against import statements found in SAST findings' file paths and evidence. This provides ~50% noise reduction with minimal complexity.

**Phase 2 (v0.3.0)**: CodeQL-powered — use CodeQL's dataflow analysis to trace actual call paths to vulnerable library functions. Requires building a combined code+dependency graph.

### Integration Point

In `pipeline/analysis.py`, after normalization and dedup, before synthesis:

```python
# After dedup

# NEW: Reachability filter
reachability = ReachabilityAnalyzer()
findings = reachability.analyze(findings)
```

### Settings Resolution Chain

The pipeline resolves settings in this priority order:

```
1. UserSettings (per-user, if user_id available) → highest priority
2. AppConfig (global admin overrides) → fallback
3. Environment variables (Settings class) → default
```

**Updated resolution in `pipeline/analysis.py`**:

```python
async def _resolve_api_keys(
    db: AsyncSession, settings: Settings, secret: str, user_id: uuid.UUID | None = None
) -> dict[str, str]:
    """Resolve LLM API keys: UserSettings → AppConfig → env."""
    api_keys: dict[str, str] = {}
    key_names = ["llm_openai_api_key", "llm_gemini_api_key", "llm_github_api_key", "llm_anthropic_api_key"]

    # Layer 1: Per-user overrides (highest priority)
    user_overrides: dict[str, str] = {}
    if user_id is not None:
        result = await db.execute(
            select(UserSettings).where(
                UserSettings.user_id == user_id,
                UserSettings.key.in_(key_names),
            )
        )
        for row in result.scalars().all():
            decrypted = decrypt_value(json.loads(row.value), secret)
            if decrypted:
                user_overrides[row.key] = decrypted

    # Layer 2: Global AppConfig overrides
    result = await db.execute(select(AppConfig).where(AppConfig.key.in_(key_names)))
    global_overrides = {r.key: r for r in result.scalars().all()}

    # Layer 3: Env defaults — merge in priority order
    for k in key_names:
        if k in user_overrides:
            api_keys[k] = user_overrides[k]
        elif k in global_overrides:
            decrypted = decrypt_value(json.loads(global_overrides[k].value), secret)
            if decrypted:
                api_keys[k] = decrypted
                continue
        env_val = getattr(settings, k, None)
        if isinstance(env_val, SecretStr) and env_val.get_secret_value():
            api_keys[k] = env_val.get_secret_value()

    return api_keys
```

### New API: User Settings (`api/user/settings.py`)

```
GET    /api/user/settings           → list current user's settings (secrets masked)
PUT    /api/user/settings           → update current user's settings (partial update)
DELETE /api/user/settings/{key}     → delete a user setting override (falls back to global)
```

### New API: User Repositories (`api/user/repositories.py`)

```
GET    /api/user/repositories                  → list current user's repos
POST   /api/user/repositories                  → add a new repo
GET    /api/user/repositories/{id}             → get repo details
PUT    /api/user/repositories/{id}             → update repo config

```

All endpoints enforce ownership: a user can only access their own repos. `user_id` is always derived from `get_current_user`, never from request body.

### Isolation Guarantees

1. **Query-level isolation**: All queries filter by `user_id = current_user.id`
2. **No cross-user enumeration**: Repository/settings endpoints never accept a `user_id` parameter
3. **Cascading delete**: If a user is deleted, all their settings and repos are deleted (`ondelete="CASCADE"`)
4. **Secret encryption**: User API keys use the same `encrypt_value`/`decrypt_value` as global settings
5. **Masked display**: API responses never return plaintext secrets — always `mask_secret()`
