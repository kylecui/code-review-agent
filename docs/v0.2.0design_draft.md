这是一份经过系统化整理、面向生产环境的开源融合静态代码扫描平台（Open-Source Fusion SAST Platform）设计方案。该方案旨在通过整合顶级开源工具，以零授权成本实现商业软件（如 Fortify/Checkmarx）约 70% 的核心能力。
------------------------------
# “天眼”开源融合代码安全平台设计方案

## 1. 核心设计哲学

* 多维融合：单一工具必有盲点。通过“模式匹配（广度）+ 语义分析（深度）+ 关联分析（精度）”三层叠加，构建防御体系。
* 双轨并行：增量拦截（免疫系统，确保未来安全）与存量清扫（手术清创，消化历史债务）同步进行。
* 标准化输出：强制所有工具统一采用 SARIF (Static Analysis Results Interchange Format) 协议，实现跨引擎的数据聚合。

------------------------------
## 2. 总体架构设计
平台分为四层：接入层、执行层、中枢处理层、治理层。
## 2.1 接入层 (Ingress)
支持 Webhook 挂载，适配 Git 提交、Pull Request (PR) 触发以及定时全量扫描任务。
## 2.2 执行层 (Multi-Engine Pool)
根据扫描目标自动调拨引擎，分为三个层级：

* L1：快检层 (Pattern Matching)
* Semgrep OSS: 核心引擎，负责跨语言的 OWASP Top 10 基础规则匹配。
   * Gitleaks / TruffleHog: 专项扫描硬编码密钥、API Token。
* L2：语义深度层 (Deep Semantic)
* Java: SpotBugs + FindSecurityBugs (字节码分析)。
   * Go: Golangci-lint (集成 Gosec)。
   * Python: Pysa (Facebook出品，强污点分析)。
   * C/C++: Cppcheck + Clang-Tidy。
* L3：逻辑分析层 (Graph-based Analysis)
* CodeQL CLI / Joern: 构建代码属性图 (CPG)，进行跨文件的全路径污点追踪（Taint Tracking）。

## 2.3 中枢处理层 (The Brain - 核心差异化竞争力)
这是实现“70% 商业能力”的关键：

* SARIF 适配器：将各引擎输出的异构 JSON 统一化。
* 指纹算法 (Fingerprinting)：计算 Hash(RuleID + File_Path + Normalized_AST_Snippet)。忽略行号，确保代码重构后误报标记依然生效。
* 可达性分析 (Reachability Analysis)：将 SCA（依赖漏洞）结果与 SAST（代码调用）结果对撞。只有当代码真实调用了受损库的危险函数时，才提升漏洞等级。

## 2.4 治理层 (Management)

* DefectDojo: 漏洞全生命周期管理，支持去重、分级、指纹库存储。
* 看板与通知: 集成 Jira、钉钉/企微，实现“检出-派单-修复-闭环”。

------------------------------
## 3. 运行策略设计：双轨模型

| 特性 | 增量轨道 (PR 扫描) | 存量轨道 (全量审计) |
|---|---|---|
| 触发时机 | 代码提交 / PR 创建 | 每周末或版本发布前 |
| 运行引擎 | Semgrep + 密钥扫描 + SCA | 全引擎 (含 CodeQL / Joern) |
| 时延目标 | < 3 分钟 (Pipeline 级) | 无硬性要求 (通常 1-4 小时) |
| 处理动作 | 发现高危直接 Block (阻断) | 自动同步至 Jira 看板 |
| 业务价值 | 止损，确保不产生新漏洞 | 清障，降低生产环境风险 |

------------------------------
## 4. 实施路径图 (Roadmap)## 第一阶段：MVP 构建 (1-4周) —— 建立“地基”

   1. 基础设施：部署 DefectDojo 及数据库。
   2. 核心引擎集成：封装 Semgrep 与 Trivy 为标准 Docker 镜像。
   3. 结果打通：实现“扫描-转化SARIF-导入Dojo”的全流程。

## 第二阶段：免疫系统上线 (5-8周) —— 建立“防火墙”

   1. CI 集成：在公司主流 GitLab/GitHub 流水线中嵌入扫描插件。
   2. 基线策略：对存量漏洞进行“一键忽略”，设定“增量零容忍”规则。
   3. 反馈闭环：实现 PR 评论区自动回贴，直接告知开发者修复建议。

## 第三阶段：深度治理 (9-12周) —— 攻坚“逻辑漏洞”

   1. 引入 CodeQL：针对 Java、Go 等核心业务语言编写自定义污点查询规则。
   2. SCA + SAST 联动：上线可达性分析模块，剔除 50% 以上的“无意义”依赖漏洞警告。
   3. 规则精简：根据历史误报率，关停高误报规则，沉淀公司级安全编码规范。

------------------------------
## 5. 关键成功指标 (KPIs)

* 误报率 (FPR)：调优后需低于 20%（通过指纹库和语义过滤实现）。
* 漏报率 (FNR)：通过 OWASP Benchmark 靶场测试，核心漏洞（注入、越权）检出率需 > 60%。
* 接入覆盖率：核心业务线 100% 接入 CI 增量扫描。
* 修复时效：高危漏洞从发现到关闭的平均周期控制在 72 小时内。

------------------------------
## 6. 维护与运营建议

* 安全工程师职责：每周 Review 一次“误报 Top 10”，通过更新 Semgrep 忽略规则或调整 CodeQL 查询逻辑来持续降噪。
* 开发者激励：在 DefectDojo 中建立“代码安全先锋榜”，表彰修复漏洞积极的团队，从“要我改”转变为“我要改”。

------------------------------
结语：本方案不追求昂贵的工具堆砌，而追求数据的精准流动与工程化闭环。通过这套体系，团队可以构建起一道自动化、可进化的安全防线。
